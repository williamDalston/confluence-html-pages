Dimension Tables

This page documents our standard dimension table structures. Dimension tables provide descriptive context for the measures in fact tables.

Dimension Table Principles

  • Contain descriptive attributes and hierarchies
  • Have a single primary key (surrogate key recommended)
  • Should be denormalized for query performance
  • Include slowly changing dimension (SCD) logic when needed

Standard Dimension Tables

Dim_Date

  • Comprehensive date dimension with calendar hierarchies
  • Attributes: Year, Quarter, Month, Week, Day, Fiscal Year, etc.
  • Standard across all models for time-based analysis

Dim_Content

  • Content metadata and attributes
  • Attributes: Content ID, Title, Type, Category, Author, Publish Date
  • Links to Drupal content management system

Dim_Team

  • Organizational structure and team information
  • Attributes: Team ID, Name, Department, Manager, Location
  • Used for Jira and workflow analytics

Dim_Ticket

  • Jira ticket attributes and metadata
  • Attributes: Ticket Key, Type, Priority, Status, Component, Labels
  • Links to Fact_TicketHistory

Best Practices

  • Always include a Date dimension in every model
  • Use surrogate keys (integer IDs) rather than natural keys
  • Include descriptive columns for reporting and filtering
  • Maintain consistent naming conventions across models
